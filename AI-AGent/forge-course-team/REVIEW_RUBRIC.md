# Review Rubric

Use this before accepting any chapter plan or lesson.

## 1. Zero-base safety

Reject or rewrite if:
- it assumes the learner reads code fluently
- it introduces multiple abstractions without isolation
- it uses terminology before grounding it
- it compresses explanations because the writer is already expert

## 2. Concept load

Reject or split if:
- more than one major new idea is introduced at once
- more than 3 fresh terms appear in the same beginner lesson
- syntax and semantics both change sharply in the same step

## 3. Block fit

Reject or revise if:
- early lessons rely too heavily on fill-in or free production
- code-runner appears before syntax confidence exists
- match-blocks is used before parts are known
- multiple-choice explanations are weak or generic

## 4. Misconception handling

Reject or revise if:
- the lesson never surfaces likely wrong mental models
- distractors are random instead of novice-plausible
- the explanation says only “wrong” without explaining why

## 5. Pacing

Reject or split if:
- the lesson feels like a lecture with token exercises
- checkpoints are too sparse
- difficulty ramps inside the lesson are abrupt
- the learner must hold too many symbols in working memory

## 6. Language quality

Reject or rewrite if:
- wording sounds like a textbook
- sentences are too compressed
- tone is patronizing or showy
- examples are abstract when concrete ones are available

## 7. Large-scale curriculum quality

Reject or revise chapter plans if:
- chapters are broad enough to hide difficulty spikes
- “data types”, “operators”, or “conditionals” are treated as single lessons for beginners
- there is no reinforcement spacing
- there is no prerequisite map

## 8. Acceptance bar

Accept only when the output feels like it was designed by:
- a curriculum architect
- a novice cognition specialist
- a practice designer
- a strict quality editor

If it only feels like “an AI that can write lesson code”, it is not good enough.
